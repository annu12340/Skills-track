The mechanics behind the Rescue Rangers

A plain-language deep dive into how each skill actually thinks — the exact commands it runs, the decisions it makes, and the moment it blocks, finds, or fixes the thing that would have wrecked your week.

Prevent · The Bouncer
Semantic Commit Guard

"Lint checks form. This checks meaning — and lies."

Linters and pre-commit check form: syntax, formatting, style. They can't tell that a line means "production password," or that logic landed in the wrong layer, or that your README started contradicting your code. This skill is the missing judgment layer. It reads the staged diff and reasons about what the change actually does.

The core trick

It only ever looks at the staged snapshot (git diff --cached) — the exact thing about to become a commit — never your messy working tree. It judges what you're really committing, nothing more.

Two modes, same checks

There are really two guards. The on-demand review is the agent reading your diff and reasoning about it. The optional installed hook is a fast, deterministic script that catches the crude disasters automatically on every commit. They're a team — neither replaces the other.

How the on-demand review works

It follows a fixed five-step workflow. Here's what it actually runs:

  1. Is there anything to review? Runs git diff --cached --stat and finds the repo root. If nothing's staged, it stops — there's nothing to guard.
  2. Load your policy. This is the key step. It looks for your rules in priority order: .semantic-guard.md, then AGENTS.md/CLAUDE.md/CONTRIBUTING.md, then lint configs, then built-in defaults. A written rule like "handlers must not touch the DB" is what turns an opinion into an enforceable verdict.
  3. Collect evidence with exact line numbers. Uses git diff --cached -U0 to strip context so every added line maps to a precise file:line it can cite.
  4. Run the four checks over every changed line (below).
  5. Deliver a verdict + receipt. Leads with PASS/BLOCK/WARN, lists each finding with a concrete fix, writes a Rescue Receipt, then stops. It never edits, unstages, or commits — blocking means reporting; you stay in control.

The four checks

CheckWhat it huntsSeverity
SecretsAPI keys, tokens, passwords, private keys, inline .env valuesBLOCK if plausibly live
ArchitectureViolations of your loaded policy — logic in the wrong layer, banned imports, bypassed abstractionsBLOCK w/ policy · WARN if a hunch
Docs-syncDocstrings, READMEs, or comments that no longer match the changed behaviorWARN
Risk smellsDebug prints, commented-out code, disabled tests, broad exception swallowing, accidental large filesWARN · BLOCK for secret files
The decision logic — why this matters

Secrets are the one hard block — flagged even on suspicion, always redacted, always "rotate it." Architecture only blocks with a written policy; without one it's downgraded to a warning so the guard never bullies you with its own taste. And a key in tests/fixtures/ or an obvious dummy is a warning, not a block.

What a verdict looks like

GUARD: BLOCKED — 2 blocking issues, 3 warnings BLOCK app/config.py:1-2 [secrets, high] Hardcoded API key + DB password replacing env reads: <redacted> Fix: restore os.environ reads — and rotate the exposed credentials. BLOCK app/handlers.py:1,5-14 [architecture, policy] Raw SQL + business logic in the thin handler. Policy: only service.py may touch the DB. Fix: delegate back to service.get_user(). WARN README.md:7-10 [docs-sync] "No secrets are stored… safe to make public" now contradicts config.py.

The installed hook — and the division of labor

Run scripts/install-hook.sh and it drops a script into .git/hooks/pre-commit. On every commit, git runs it — but it's deliberately dumb and fast: a deterministic secret + private-key + large-file scan that exits nonzero to block, with no thinking involved. It then reminds you to run the full review.

  • The hook is the seatbelt that's always on — but it can only catch crude, pattern-matchable disasters.
  • The agent review is the one that understood your handler swallowed the database layer and that your README started lying — because it read the diff against rules you wrote down.
Important pitfall

A leaked secret isn't fixed by unstaging. Once it's written to a file you control — and especially once committed — it must be rotated. git reset does not make it safe.

Diagnose · The Detective
Git Bisect AI

"Turn 'this broke last week' into 'commit a1b2c3d broke it, here's why.'"

Somewhere in the last 200 commits, something that worked stopped working. git bisect can binary-search history to find the exact culprit — testing log₂(N) commits instead of all N — but it needs a reliable way to ask each commit "are you broken?" This skill acts as a forensic QA engineer: it pins down the bounds, builds that reliable test, drives the search unattended, then explains the offending change.

The make-or-break

The whole technique lives or dies on one thing: a reproduction command that reliably exits 0 when the bug is absent and nonzero when it's present. Get that right and the rest is mechanical. Most of the effort goes here — a flaky test produces a confident, wrong answer.

How it works, step by step

  1. Safety check. Runs git status --porcelain and records the current branch + SHA so it can always put you back. If your tree is dirty, it bisects in a disposable git worktree instead of asking you to stash unrelated work.
  2. Define the bug and pick bounds. A bad commit (usually HEAD) and a good commit where it worked. It translates hints — "last week" becomes git rev-list -1 --before="7 days ago" HEAD; "the 1.2 release" becomes the tag.
  3. Choose a reproduction. A single failing test, a build command, or a headless browser script — whatever exits nonzero on the failure. The narrowest test bisects fastest.
  4. Validate the repro (never skipped). It proves the command fails on the bad commit and passes on the good one before searching. If not, the bounds or repro are wrong — fixed here, not after wasting a full bisect.
  5. Run the automated search. git bisect run with the bundled runner checks out each midpoint, runs your test, and narrows until it names the first bad commit.
  6. Report, then always clean up. Runs git bisect reset to restore HEAD (even on failure), then reads the culprit's diff and explains why that specific change caused the failure — not just that it's the boundary.

The runner's exit-code contract

The bundled bisect-run.sh wrapper translates your test's result into git bisect's verdict. This is the clever part — it makes the search robust against a test command that misbehaves:

Test exit codeVerdict to gitMeaning
0goodThe bug is absent here.
125skipCan't judge this commit (e.g. it won't build while hunting a runtime bug).
anything elsebadThe bug is present. Codes that could abort the search are collapsed to plain "bad" so a misbehaving test can't derail the run.

Handy flags adapt it to other hunts: --build-is-bug (a failing build is the regression), --invert (find when a failing test started passing), and --timeout to bound each step.

A worked example

A multiply(a, b) test regressed; the user says "math was fine in v1.0."

# validate the repro discriminates BEFORE bisecting pytest tests/test_maths.py::test_multiply -q # on HEAD: FAILS (bug present) git checkout v1.0 && pytest …::test_multiply -q # PASSES (bug absent) # run it unattended git bisect start HEAD v1.0 git bisect run "$RUNNER" --test "pytest tests/test_maths.py::test_multiply -q" → a1b2c3d is the first bad commit git bisect reset && git show a1b2c3d # the diff: multiply() was changed to `return a + b`

The report names commit a1b2c3d, its author/date/message, and the why: the operator was changed from * to +, so multiply adds instead of multiplies.

Important pitfalls

The "good" commit isn't always good — a bad lower bound produces a confident-but-wrong culprit, which is why validation is mandatory. And the repro script lives outside the checkout (e.g. in $TMPDIR) so historical checkouts can't delete it mid-search.

Recover · The Medic
Dependency Upgrade Loop

"Bumping one framework cascades through everything. This works the cascade methodically."

A major upgrade rarely fails in one place — bumping a framework cascades through peer dependencies, renamed APIs, removed exports, and changed defaults. Try to fix it all at once and you get an unreadable wall of errors with no clear cause. This skill works the cascade one logical change at a time: bump, build, read the first error, fix exactly that, rebuild — looping until green or until it hits a real boundary worth reporting.

The discipline that makes it work

One logical change per verify run. Sometimes that's one package; sometimes it's a framework plus its required peers. Keep each step small enough that the next failure has a clear, single cause.

How it works, step by step

  1. Baseline. Records git status --porcelain, the package manager, and runtime versions, then runs the verify command once before changing anything — so "green" later means something. It preserves your work and never commits or removes lockfiles unasked.
  2. Inventory + ordered plan. Parses the manifest and orders the work safest-first: patch/minor bumps, then the framework everything pegs to, then that framework's peers, then everything else.
  3. Read the migration notes first. Knowing componentWillMount → useEffect or url() → re_path() up front turns a blind loop into a targeted edit. It uses official release notes, not model memory.
  4. The loop. For each item: apply one bump → install → build with the probe → find the first root error → fix exactly that → rebuild until green → checkpoint.
  5. Report. A package: old → new table, the code migrations made, any behavior changes that need human review, and the final verification output proving green.

The build probe is the oracle

Every iteration runs through scripts/build-probe.sh, which executes your verify command, saves the full log, and highlights the likely root-error lines. The verify command is the loop's pass/fail signal — it must exit 0 when healthy and nonzero when broken.

"$PROBE" --cmd "npm run build" --timeout 600 --log /tmp/dep-upgrade.log → root error: Module '"react-dom/client"' has no exported member 'render' (later errors are usually cascades of this one — fix the first)
The loop guard — why it won't thrash

If the same root error survives 3 focused fix attempts, it stops and reports a boundary instead of thrashing. It will not silence the problem with --legacy-peer-deps, transitive pins, or @ts-nocheck to fake a green build — those are reported as temporary boundaries, never as the fix.

Triaging failures accurately

FailureWhat it does
Resolver / peer conflict (ERESOLVE)Solves the version graph first — never bypasses with force flags.
Removed export / importFinds the replacement API, updates the smallest affected call sites, rebuilds.
Type errors after a majorChecks whether the library changed public types or local code leaned on an internal one.
Runtime failures after compileCompares migration notes for new defaults, stricter parsing, async/auth changes.
Important distinction

Behavior changes are decisions, not syntax fixes. A migration that compiles can still change runtime semantics — a new default, stricter parsing, a dropped feature. These get flagged explicitly for human review rather than waved through. And "it compiles" isn't "it works" — the test suite is preferred as the oracle whenever one exists.
