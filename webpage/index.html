Semantic Commit Guard

"Lint catches typos. This catches intent — and lies."

Why you need it

Linters and pre-commit hooks can catch syntax errors and formatting issues, but they cannot reason about intent. This skill analyzes changes before they become commits, identifying leaked credentials, architectural violations, risky code patterns, and contradictions between documentation and implementation. It prevents costly mistakes from ever reaching the main branch.

See how it works

  1. Agent reads your staged diff before commit finalizes
  2. Scans for secret patterns, architecture violations, doc/code contradictions
  3. Rejects with exact line numbers + fix guidance
  4. You fix it. Main stays innocent. Weekend preserved.
Example 1 — Layer Violation
# routes/checkout.py checkout(): payment_gateway.charge(order.total) # ARCH.md "Route handlers must remain thin and delegate all business logic to service classes."

The guard reads both files, spots that business logic has been embedded in the handler layer, and blocks the commit before the architecture starts to drift.

Skill II · Diagnose · The Detective
skills/git-bisect-ai/

Git Bisect AI

"Forty-seven commits. One culprit. Five minutes. Zero blame roulette."

Why you need it

Somewhere in the last 47 commits, a bug was born.

Instead of turning developers into amateur archaeologists, Git Bisect AI automates the entire investigation.

It validates the reproduction, performs a binary search through history, identifies the exact culprit commit, and explains why that change broke production.

See how it works

  1. Confirm good commit, bad commit, and a reproduction command (exit 0 = good, nonzero = bad)
  2. Validate the repro on both bounds — garbage in, garbage out
  3. Run git bisect run with the included bisect-run.sh wrapper
  4. Report culprit SHA, full diff, and why that change broke things
  5. Always reset bisect state. Always.
Standalone scenario · bisect-demo
# reproduce command multiply(4, 5) # expected 20 # bad commit change return a + b # should be a * b # good/bad bounds good: v1.0 bad: HEAD

The agent binary-searches history in a handful of steps, identifies the exact bad commit, and explains why the regression happened. No guard. No deps. Pure detective work.

Dependency Upgrade Loop

"Old dependency stacks fail one boundary at a time. Fix them the same way."

Why you need it

Major upgrades cascade. Bump one package → three peer deps scream → import paths moved → you're reading changelogs at 2 AM. Most upgrade attempts die at the first compile error. This skill treats migration as a loop: build, fail, fix, retry, repeat.

See how it works

  1. Parse manifests (package.json, requirements.txt, etc.)
  2. Cross-reference changelogs, upgrade notes, error databases
  3. Run build → analyze failure → rewrite affected code → bump deps → retry
  4. Continue until build passes or upgrade boundaries are identified
Python · pip / poetry Node · npm / yarn / pnpm Go · go mod Rust · cargo Ruby · bundler Java / Kotlin · Maven / Gradle
Standalone scenario · pydantic v1 → v2 (Python)
# requirements.txt Flask==1.1 pydantic==1.10 requests==2.22 # code import from pydantic import BaseSettings # ← moved to pydantic-settings in v2 # target upgrade pydantic v2 moves BaseSettings to pydantic-settings .dict() renamed to .model_dump()

The loop bumps pins, adds the missing package, rewrites imports and method calls, retries until the build passes — and flags the silent ValidationError format change as a behavior decision. Works the same way for Go modules, Rust crates, and npm packages.

Questions you might ask

How is this different from regular linting or CI?

Linters check syntax and style. CI runs after the fact. Semantic Commit Guard evaluates the meaning of changes — catching doc/code contradictions and security intent violations that no regex rule covers. Git Bisect AI and Dependency Upgrade Loop go further: they perform multi-step autonomous workflows that would take an engineer hours manually.

Do I need to install anything?

No. Skills are plain Markdown in skills/ — copy the folders into .cursor/skills/, .claude/skills/, or .codex/skills/ and your agent picks them up automatically. Each skill also ships a helper script the agent drives — bisect-run.sh, install-hook.sh, and build-probe.sh — resolved automatically at runtime.

Can these work outside Cursor?

Yes — and Cursor isn't even the primary target. These skills follow the open Agent Skills format. The same SKILL.md works in Claude Code (.claude/skills/), Cursor (.cursor/skills/), Codex (.codex/skills/), and any other tool that reads the format. Use bash scripts/install.sh to auto-detect the right directory.

How is dependency-upgrade-loop different from Renovate or Dependabot?

Renovate and Dependabot open upgrade PRs automatically and silently. They don't read changelogs, don't rewrite call sites for new APIs, and don't flag when a "passing" upgrade changes runtime behavior. dependency-upgrade-loop is an on-demand reasoning loop: it reads migration notes before bumping a major version, fixes the first root error (not the downstream cascades), and explicitly surfaces behavior changes as decisions for you to review. Use both — they solve different problems.

Why three separate demo repos instead of one?

Because each skill needs a starting state that contradicts the others. Bisect wants a clean history that passes at the good commit; the guard wants a dirty, staged diff to block; dependency-upgrade-loop wants a build that's broken from the start. Cram them into one repo and a visitor can't tell which "brokenness" belongs to which skill — and the dependency mess pollutes the bisect history. So they live as isolated sandboxes under one umbrella repo: single clone, one top-level README, three independent demo dirs, each reset by its own setup.sh.
